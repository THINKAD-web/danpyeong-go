# 초등학교 단원평가 앱 — 기획서

> 하이브리드(학생 자습형 + 교사 출제형) · PWA 우선
> 작성: 2026.05 · 담당: THINKAD

---

## 1. 한 줄 요약

교육과정 단원별 **문제은행 코어**를 공유 자산으로 두고, 그 위에 **학생 자습형**과 **교사 출제형** 두 진입점을 얹는 하이브리드 학습 평가 서비스. 웹앱(PWA)으로 빠르게 출시한다.

---

## 2. 타깃 & 모드

| 모드 | 사용자 | 흐름 | 수익 |
|------|--------|------|------|
| 학생 자습형 (B2C) | 학생 / 학부모 | 미리 채워둔 문제은행에서 단원 골라 응시 | 콘텐츠 구독, 광고 |
| 교사 출제형 (B2B) | 학교 / 학원 교사 | 같은 문제은행에서 시험지 조립 → 학급 배포 | 구독 (학급/학교 단위) |

**핵심 이점:** 교사 출제형은 자습형 문제은행을 재활용한다. 콘텐츠를 한 번 만들면 양쪽에서 모두 쓰이므로 제작 효율이 가장 높다.

---

## 3. 설계 원칙

처음부터 두 모드를 동시에 만들면 무너진다. **공통 코어 → 두 개의 얼굴** 순서로 쌓는다.

- 공유 코어: 문제은행 · 응시 엔진 · 자동채점 · 결과 리포트
- 위에 얹는 레이어: (학생) 추천·진도 / (교사) 시험 조립·배포·성적 분석
- 차별화 포인트: **단원별 취약점 리포트** — 단순 점수가 아니라 "어느 단원의 어떤 개념이 약한지" 짚어주는 분석

---

## 4. 핵심 기능 (우선순위순)

1. 학년·과목·단원 트리 구조의 문제은행
2. 문항 타입: 객관식 / 단답 / 서술
3. 자동채점 (객관식·단답) + 서술형은 수동/AI 보조
4. 응시 화면: 타이머, 진행률, 임시저장
5. 즉시 결과 + **오답노트**
6. 단원별·학생별 취약점 리포트
7. (교사) 학급 관리, 시험 조립, 배포, 성적 대시보드
8. (관리자) 문제은행 등록 어드민

---

## 5. DB 스키마 (뼈대)

처음에 이 구조만 잘 잡으면 확장해도 무너지지 않는다.

```
Subject (과목) ─ Unit (단원) ─ Question (문항)
                                  ├ type: 객관식 / 단답 / 서술
                                  ├ difficulty, 정답, 해설
                                  └ Choice[] (선택지)

User (role: 학생 / 교사 / 관리자)
Classroom (학급) ─ Enrollment (학생 소속)
Assignment (교사가 배포한 시험) ─ AssignmentQuestion[]
Attempt (응시 기록) ─ Answer[] ─ 자동채점 결과
```

**포인트:** `Question`을 `Assignment`(시험)와 분리한다. 한 문항이 자습용으로도, 여러 교사의 시험지에도 동시에 쓰여야 하기 때문. 시험은 문항을 "참조"할 뿐 복제하지 않는다.

---

## 6. 화면 구성 (IA)

- **공통:** 로그인 / 학년·과목·단원 선택 / 응시 화면 / 결과·오답노트
- **학생 모드:** 홈(추천 단원·진도) → 단원 선택 → 응시 → 리포트
- **교사 모드:** 대시보드 → 시험 만들기(문항 담기) → 학급 배포 → 성적·취약점 분석
- **관리자:** 문제은행 등록 어드민 *(없으면 운영 자체가 불가)*

---

## 7. 기술 스택

| 영역 | 선택 |
|------|------|
| 프레임워크 | Next.js (App Router) |
| DB / ORM | Neon (PostgreSQL) + Prisma |
| 인증 | NextAuth (역할 기반: 학생/교사/관리자) |
| 배포 | Vercel |
| PWA | next-pwa (홈 화면 추가, 오프라인 캐싱) |
| 스타일 | Tailwind CSS |
| AI | Claude API (문항 생성·서술형 채점 보조) |

**웹앱(PWA) 우선 근거:** 네이티브 앱의 스토어 심사·양대 OS 유지보수 부담 없이 빠르게 출시. 초등 단원평가는 태블릿·PC 사용 비중이 높아 웹이 유리하다.

## 8. 단계별 로드맵

### Phase 1 — 코어 + 학생 자습형 (MVP)
- 문제은행 스키마, 어드민 입력 도구
- 응시·자동채점, 오답노트
- **검증 범위:** 한 학년·한 과목만 먼저 채움 (예: 6학년 수학)

### Phase 2 — 리포트 강화
- 단원별 취약점 분석, 학습 진도 추적
- 서비스의 핵심 차별화 지점

### Phase 3 — 교사 출제형 레이어
- 학급/학생 관리, 시험 조립·배포, 교사 대시보드
- 코어 위에 얹기

### Phase 4 — PWA + 확장
- 홈 화면 추가, 오프라인 캐싱
- 과목·학년 확대

## 9. 운영 전략 — 진짜 중요한 것

> 성패는 **개발이 아니라 콘텐츠·저작권**에서 갈린다.

- **저작권:** 교과서·기출 문제를 그대로 쓰면 안 된다. **자체 제작 필수.**
- **교육과정:** 2022 개정 교육과정에 맞춰 단원·성취기준을 구성한다.
- **콘텐츠 업데이트:** 꾸준한 문항 추가·갱신이 서비스 생명력을 좌우한다.
- **AI 문항 생성 어드민(Phase 1에 포함 권장):**
  단원 입력 → Claude가 문항 초안 생성 → 사람이 검수·확정.
  콘텐츠 수급 속도를 크게 끌어올려 운영 부담을 줄인다.
